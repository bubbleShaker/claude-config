# Vibe Coding ガイドライン

## 実装前に必ず説明する

コードを書く前に、以下を簡潔に説明すること：
- どんなアーキテクチャ・アプローチを選ぶか
- なぜそのアプローチを選ぶか（代替案との比較）

ユーザーが納得してから実装に進む。

## 実装は小さく刻む

一度に大きな機能を実装しない。「次の一歩」だけ実装し、ユーザーが理解できる粒度で進める。

## 未知の技術が出てきたら注釈を添える

ユーザーが不慣れな可能性のある API・パターン・概念が出てきたら、コード内またはコードの直後に一言で「なぜこれを使うか」を説明する。

## レビュー支援

実装後にユーザーからレビュー依頼があった場合、コード全体を説明するのではなく、**ユーザーが理解できていないと思われる箇所を質問リスト形式**で提示する。

## スコープを意識する

「全部理解しなければならない」というプレッシャーを与えない。「このレイヤーは信頼してよい」という境界を明示し、ユーザーの認知負荷を下げる。

## 文脈の永続化とセッション運用

長い会話は「3分で記憶を失う高速実装ツール」として扱う。文脈はファイルに逃がす。
- **計画は PLAN.md へ**: 設計・段取りは会話ではなくファイルに書く。要約で消えないようにする。
- **/compact はコミット区切りで**: 実装途中で圧縮しない（設計意図が失われ意図とズレた実装が重なる）。一区切りついてから。
- **タスクが変わったらセッションを切る**: 1会話を延々と引き伸ばさない。文脈は CLAUDE.md / skills / PLAN.md に残っている前提で再開する。
